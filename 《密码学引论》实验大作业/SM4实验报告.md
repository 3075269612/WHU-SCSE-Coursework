# 一、实验目的

密码学是保障信息安全的核心技术。分组密码作为一种重要的对称密码体制，在数据加密、身份认证、消息认证等领域有着广泛的应用。SM4算法是我国国家密码管理局于2006年发布的分组密码标准算法（GM/T 0002-2012），其安全性和实现效率已得到充分验证。

本实验旨在以SM4分组密码算法为基础，设计并实现一个完整的文件加密软件系统。通过本实验，达到以下目的：

1. **深入理解SM4分组密码算法**：掌握SM4算法的加密流程、密钥扩展算法、轮函数结构等核心内容，理解分组密码的设计原理。

2. **掌握分组密码工作模式**：重点掌握密文分组链接（Cipher Block Chaining, CBC）工作模式的原理与实现方法，理解其在安全性方面的优势。

3. **掌握短块处理技术**：学习并实现密文挪用（Ciphertext Stealing, CTS）技术，解决分组密码处理任意长度数据时的短块问题，理解CTS与PKCS#7等传统填充方案的本质区别。

4. **掌握混合加密体制**：理解对称密码与非对称密码各自的优势与局限，掌握RSA+SM4混合加密方案的设计与实现，解决对称密钥的安全分发问题。

5. **培养工程实践能力**：综合运用C语言编程、OpenSSL密码库、GTK图形界面框架等技术，完成从理论到实践的完整工程实现。

6. **学习性能评估方法**：通过设计合理的性能测试方案，评估加密系统的吞吐量、时延等关键性能指标，理解影响密码系统性能的关键因素。

# 三、密码学基础理论

本章系统阐述本实验所涉及的核心密码学理论，包括SM4分组密码算法、CBC工作模式、密文挪用技术、RSA公钥密码算法以及混合加密体制。

## 3.1 SM4分组密码算法

### 3.1.1 算法概述

SM4是国家密码管理局于2006年1月公布的我国商用密码分组密码标准算法，标准编号为GM/T 0002-2012《SM4分组密码算法》。该算法于2016年成为ISO/IEC国际标准（ISO/IEC 18033-3:2010/Amd 1:2021）。SM4与AES-128具有相同的安全强度级别，广泛用于WAPI无线局域网安全标准、金融IC卡、电子政务等领域。

SM4算法的主要参数如下：

| 参数 | 值 |
|------|-----|
| 明文分组长度 | 128 bit (16 byte) |
| 密钥长度 | 128 bit (16 byte) |
| 轮数 | 32 轮 |
| 轮函数结构 | 非平衡Feistel结构 |
| S盒规模 | 8×8 S盒（与AES S盒不同的固定置换） |
| 安全强度 | 约128 bit（抵抗已知最优攻击） |

### 3.1.2 加密流程

SM4加密算法将128 bit明文分为4个32 bit字 $(X_0, X_1, X_2, X_3)$，经过32轮迭代变换后输出4个32 bit密文字，最后进行反序变换得到密文。

**轮函数定义**：

对于第 $i$ 轮（$i = 0, 1, \ldots, 31$）：

$$X_{i+4} = X_i \oplus T(X_{i+1} \oplus X_{i+2} \oplus X_{i+3} \oplus rk_i)$$

其中：
- $rk_i$ 是第 $i$ 轮的32 bit轮密钥，由密钥扩展算法生成
- $T$ 是合成置换：$T(x) = L(\tau(x))$
- $\tau$ 是S盒替换：将32 bit输入分为4个字节，每个字节经8×8 S盒替换后重新组合
- $L$ 是线性变换：$L(B) = B \oplus (B \ll 2) \oplus (B \ll 10) \oplus (B \ll 18) \oplus (B \ll 24)$

**反序变换**（最后一轮之后）：

密文输出为 $(X_{35}, X_{34}, X_{33}, X_{32})$，即将最后4个状态字反向排列。

### 3.1.3 密钥扩展算法

SM4密钥扩展算法将128 bit初始密钥 $MK = (MK_0, MK_1, MK_2, MK_3)$ 扩展为32个32 bit轮密钥 $rk_0, rk_1, \ldots, rk_{31}$。

首先将初始密钥与系统参数 $FK$ 异或：

$$(K_0, K_1, K_2, K_3) = (MK_0 \oplus FK_0, MK_1 \oplus FK_1, MK_2 \oplus FK_2, MK_3 \oplus FK_3)$$

其中 $FK$ 为固定常量。

然后对于 $i = 0, 1, \ldots, 31$：

$$K_{i+4} = K_i \oplus T'(K_{i+1} \oplus K_{i+2} \oplus K_{i+3} \oplus CK_i)$$
$$rk_i = K_{i+4}$$

其中 $CK_i$ 是固定参数，$T'$ 与加密轮函数中的 $T$ 结构相同但使用不同的线性变换 $L'$。

### 3.1.4 解密流程

SM4解密流程与加密流程完全对称，唯一区别在于轮密钥的使用顺序相反：

- **加密**：依次使用 $rk_0, rk_1, \ldots, rk_{31}$
- **解密**：依次使用 $rk_{31}, rk_{30}, \ldots, rk_0$

### 3.1.5 安全性分析

SM4算法的安全性主要体现在以下几个方面：

**（1）S盒设计**：SM4的S盒基于有限域 $GF(2^8)$ 上的乘法逆元构造，具有优良的非线性性质，差分均匀度和线性偏差均很小，能有效抵抗差分密码分析和线性密码分析。

**（2）线性变换**：轮函数中的线性变换 $L$ 采用了多级循环移位与异或操作，提供了良好的扩散效果。经过3轮传播后，每个输入比特的改变可以影响到所有输出比特（完全扩散性）。

**（3）密钥扩展**：密钥扩展算法与加密轮函数结构相似，确保了轮密钥之间的独立性。固定参数 $FK$ 和 $CK$ 的引入增加了密钥扩展的非线性。

**（4）安全分析现状**：截至目前，对全轮SM4的最优攻击为不可能差分攻击（23轮）和线性攻击（23轮），远未达到32轮的全轮安全性。SM4足以抵抗所有已知的密码分析攻击。

## 3.2 分组密码工作模式

### 3.2.1 电子密码本（ECB）模式

ECB是最简单的工作模式——每个明文块独立加密，使用相同的密钥。其最大的缺点是相同明文块产生相同密文块，无法隐藏数据模式，安全性较差。下图形象地说明了ECB模式的缺陷：加密图像后仍能看出原始图像的轮廓。

### 3.2.2 密文分组链接（CBC）模式

CBC模式是本系统采用的工作模式。在CBC模式中，每个明文块在加密前先与**前一个密文块**进行异或操作，从而将块与块之间"链接"起来。其加密和解密过程如下：

**CBC加密**：

$$C_0 = E_K(P_0 \oplus IV)$$
$$C_i = E_K(P_i \oplus C_{i-1}), \quad i = 1, 2, \ldots, n-1$$

**CBC解密**：

$$P_0 = D_K(C_0) \oplus IV$$
$$P_i = D_K(C_i) \oplus C_{i-1}, \quad i = 1, 2, \ldots, n-1$$

其中 $IV$ 为初始化向量（Initialization Vector），必须随机生成且不可预测。

**CBC模式的安全性优势**：
1. **明文随机化**：每个明文块在与前一个密文块异或后被随机化，相同明文块产生不同密文块
2. **依赖性传播**：一个比特的错误在解密时会影响当前块的全部比特和下一个块的对应比特
3. **抵抗重放攻击**：由于IV的引入，相同明文加密后产生完全不同的密文

**CBC模式的"密文链接"特性**正是本实验题目中"密文链接"技术的核心体现。每个密文块既作为当前块的加密输出，又作为下一个块的加密输入，形成了链式依赖关系。

### 3.2.3 短块处理问题

分组密码要求输入数据长度为分组长度（128 bit即16 byte）的整数倍，但实际文件的长度往往不是16的整数倍。这个"短块"问题的本质是：

> 当明文长度 $L$ 不满足 $L \equiv 0 \pmod{16}$ 时，如何处理最后不足16字节的数据？

传统的处理方法有：

1. **填充法（Padding）**：在明文末尾填充若干字节，使总长度成为16的倍数。最常见的PKCS#7填充在明文末尾填充 $n$ 个值为 $n$ 的字节（$n = 1, 2, \ldots, 16$）。填充法的缺点是**密文长度大于明文长度**，产生1~16字节的膨胀。

2. **密文链接法（Ciphertext Linking）**：利用CBC模式的链接特性，将最后一个短块与前一个密文块关联处理。

3. **密文挪用法（Ciphertext Stealing）**：从倒数第二个密文块中"挪用"部分字节来填充最后一个短块，使加密后的密文总长度**恰好等于明文总长度**（零膨胀）。

本实验的核心技术要求正是同时实现第2和第3种技术。

## 3.3 密文挪用（Ciphertext Stealing, CTS）技术详解

### 3.3.1 CTS的基本思想

密文挪用技术最早在Kerberos网络认证协议（RFC 3961/3962）和NIST特别出版物SP 800-38A Addendum中被标准化。其核心思想可以用一句话概括：

> "既然最后一个明文块不够16字节，那就从倒数第二个密文块中'借'几个字节来凑齐，加密后再调整输出顺序。"

这使得加密不引入任何填充字节，密文长度与明文长度完全相等。

### 3.3.2 CTS的变体

NIST SP 800-38A Addendum定义了三种CBC-CTS变体：

| 变体 | 填充方式 | 特点 |
|------|---------|------|
| **CBC-CS1** | 从 $C^*$ 的开头取 $(16-s)$ 字节附加在 $P_s^*$ 之后 | 本系统采用 |
| CBC-CS2 | 从 $C^*$ 的末尾取 $(16-s)$ 字节附加在 $P_s^*$ 之后 | 用于Kerberos |
| CBC-CS3 | 从 $C^*$ 的开头取 $(16-s)$ 字节放在 $P_s^*$ 之前 | NIST首选 |

其中 $C^*$ 为倒数第二个明文块加密产生的临时密文块，$s$ 为最后一个不完整明文块的字节数（$1 \le s \le 15$）。

### 3.3.3 CBC-CS1加密算法

本系统实现的CBC-CS1加密算法，对于明文长度 $N \ge 16$ 字节，设 $q = \lfloor N / 16 \rfloor$，$r = N \bmod 16$：

**情况1**：$r = 0$（所有块完整）
- 执行标准CBC加密即可，无需CTS。

**情况2**：$r > 0$（存在短块）

令 $P_1, P_2, \ldots, P_{q-1}$ 为前 $q-1$ 个完整明文块，$P_q$ 为最后一个完整明文块（16字节），$P_{q+1}^*$ 为最后的不完整块（$r$ 字节）。

**步骤一（标准CBC）**：对 $P_1$ 至 $P_{q-1}$ 执行标准CBC加密，设 $C_{q-1}$ 为产生的最后一个密文块。（若 $q = 1$，则跳过此步，设 $C_{q-1} = IV$。）

**步骤二（加密倒数第二个完整块）**：

$$C_{temp} = E_K(P_q \oplus C_{q-1})$$

$C_{temp}$ 是一个完整的16字节密文块。

**步骤三（密文挪用——填充短块）**：

从 $C_{temp}$ 的**开头**取 $(16-r)$ 个字节，附加在 $P_{q+1}^*$ 之后，形成完整的16字节块：

$$P_{padded} = P_{q+1}^* \; || \; C_{temp}[0 \;..\; 15-r]$$

这一步正是"挪用"的核心——从 $C_{temp}$ 中取出的 $(16-r)$ 字节不再出现在最终输出的 $C_{temp}$ 截断部分中。被"挪用"的字节用于填充短块，而它们自身在截断的 $C_{temp}$ 输出中被丢弃。

**步骤四（加密填充后的块）**：

$$C_{last} = E_K(P_{padded} \oplus C_{temp})$$

$C_{last}$ 也是一个完整的16字节块。

**步骤五（重排输出）**：

最终密文输出为：

$$C_1 \;||\; C_2 \;||\; \cdots \;||\; C_{q-1} \;||\; C_{last} \;||\; C_{temp}[0 \;..\; r-1]$$

注意输出顺序：$C_{last}$（完整块）放在前面，$C_{temp}$ 的**前 $r$ 个字节**（而不是被挪用掉的那些字节）放在最后。密文总长度 = $(q-1) \times 16 + 16 + r = q \times 16 + r = N$，**恰好等于明文长度！**

### 3.3.4 CBC-CS1解密算法

解密过程是加密的严格逆过程。对于密文长度 $ct_len$，设 $r = ct_len \bmod 16$：

**情况1**：$r = 0$ → 标准CBC解密。

**情况2**：$r > 0$ → CTS解密。

密文结构为：$C_1 \;||\; \cdots \;||\; C_{q-2} \;||\; C_{last}(16) \;||\; C_{temp}^* (r)$，其中 $C_{temp}^*$ 是 $C_{temp}$ 的前 $r$ 字节。

**步骤一**：对 $C_1$ 至 $C_{q-2}$ 执行标准CBC解密。

**步骤二**：解密 $C_{last}$：

$$D_{last} = D_K(C_{last})$$

根据加密步骤四，$D_{last} = P_{padded} \oplus C_{temp}$，即：

$$D_{last} = (P_{q+1}^* \;||\; C_{temp}[0..15-r]) \oplus C_{temp}$$

将 $D_{last}$ 按字节展开：

$$D_{last}[i] = \begin{cases}
P_{q+1}^*[i] \oplus C_{temp}[i], & 0 \le i < r \\
C_{temp}[i-r] \oplus C_{temp}[i], & r \le i < 16
\end{cases}$$

**步骤三**（恢复短块明文）：

$$P_{q+1}^*[i] = D_{last}[i] \oplus C_{temp}^*[i], \quad 0 \le i < r$$

**步骤四**（重建完整 $C_{temp}$）：

$$
\begin{cases}
C_{temp}[i] = C_{temp}^*[i], & 0 \le i < r \\
C_{temp}[i] = D_{last}[i] \oplus C_{temp}[i-r], & r \le i < 16
\end{cases}
$$

注意到这是一个**顺序递推**的过程：对于 $i \in [r, 15]$，$C_{temp}[i-r]$ 要么来自已知的 $C_{temp}^*$（当 $i-r < r$ 时），要么来自已经在前几步恢复出的 $C_{temp}[i-r]$（当 $i-r \ge r$ 时）。

**步骤五**（恢复最后一个完整明文块）：

$$P_q = D_K(C_{temp}) \oplus C_{q-1}$$

其中 $C_{q-1}$ 为步骤一标准CBC解密时记录的最后一个输入密文块（或 $IV$，若 $q=1$）。

最终输出 $P_1 \;||\; \cdots \;||\; P_{q-1} \;||\; P_q \;||\; P_{q+1}^*$，长度 = $ct_len = N$。

### 3.3.5 CTS与PKCS#7填充的对比

| 对比维度 | CTS（密文挪用） | PKCS#7填充 |
|----------|----------------|------------|
| 密文长度 | = 明文长度（零膨胀） | > 明文长度（膨胀1~16字节） |
| 算法复杂度 | 高（需要特殊处理最后两个块） | 低（仅需添加/移除填充字节） |
| 最小明文长度 | 要求 ≥ 16字节（至少一个完整块） | 无限制 |
| 实现难度 | 较高 | 简单 |
| 标准化程度 | NIST SP 800-38A, RFC 3961/3962 | PKCS#7, RFC 5652 |
| 典型应用 | Kerberos、磁盘加密（保持扇区大小不变） | SSL/TLS、IPSec、文件加密 |

CTS特别适用于**不允许密文膨胀**的场景。例如，在磁盘扇区加密中，加密后的扇区必须与原始扇区大小相同（通常512或4096字节），此时CTS是理想的解决方案。

### 3.3.6 CTS边界情况

**（1）明文长度 < 16字节**

根据NIST SP 800-38A Addendum，CBC-CTS要求至少有一个完整明文块。对于明文长度 $< 16$ 字节的边界情况，本系统采用PKCS#7填充作为回退方案，此时密文固定为16字节。这意味着对于1~15字节的极小文件，密文会有1~15字节的膨胀。考虑到实际应用中几乎不存在如此小的文件加密需求（甚至小于SM4的密钥长度），这种回退在工程上是完全合理的。

**（2）明文长度恰好为16的倍数**

此时 $r = 0$，不需要CTS（没有短块需要处理）。直接执行标准CBC加密，密文长度依旧等于明文长度。

**（3）明文长度 > 16字节且为16的倍数+1~15**

这是CTS的标准工作场景，实现零膨胀加密。

## 3.4 RSA公钥密码算法

### 3.4.1 RSA算法原理

RSA算法由Rivest、Shamir和Adleman于1977年提出，是第一个实用的公钥密码算法。其安全性基于**大整数因子分解困难问题**。

**密钥生成**：
1. 随机选择两个大素数 $p$ 和 $q$（各约1024 bit）
2. 计算 $n = p \times q$，$\phi(n) = (p-1)(q-1)$
3. 选择公钥指数 $e$（常用65537即 $2^{16}+1$）
4. 计算私钥指数 $d = e^{-1} \bmod \phi(n)$
5. 公钥为 $(n, e)$，私钥为 $(n, d)$

**加密**：$c = m^e \bmod n$
**解密**：$m = c^d \bmod n$

### 3.4.2 本系统中的RSA使用

本系统生成**2048 bit RSA密钥对**，公钥保护在 `public.pem` 文件中，私钥保护在 `private.pem` 文件中。加密文件时使用公钥加密SM4会话密钥（OAEP填充），解密时使用私钥恢复SM4会话密钥。RSA-2048的安全性估计约为**112 bit**，足以抵抗当前的因子分解攻击。

### 3.4.3 OAEP填充

本系统使用RSA-OAEP（Optimal Asymmetric Encryption Padding）填充方案。OAEP基于Feistel网络结构，通过引入随机种子和哈希函数，实现了以下安全目标：
- **语义安全性**：相同明文每次加密产生不同密文
- **明文感知性**：攻击者无法生成有效的密文
- **抵抗选择密文攻击**：OAEP在随机预言机模型下可证明安全（IND-CCA2）

## 3.5 混合加密体制

本系统采用**RSA+SM4混合加密**方案，结合了对称密码和非对称密码各自的优势：

| 算法类型 | 在本系统中的角色 | 优势 | 劣势 |
|----------|-----------------|------|------|
| SM4（对称） | 加密文件内容 | 速度快、适合大数据 | 密钥分发困难 |
| RSA（非对称） | 保护SM4会话密钥 | 密钥分发方便 | 速度慢、不适合大数据 |

**混合加密的工作流程**：

1. **加密时**：
   - 随机生成128 bit SM4会话密钥 $K_{SM4}$
   - 使用 $K_{SM4}$ 以SM4-CBC-CTS模式加密文件内容
   - 使用RSA公钥加密 $K_{SM4}$：$C_{key} = RSA_{enc}(pub\_key, K_{SM4})$
   - 将 $C_{key}$ 和IV与密文一起存储在输出文件中

2. **解密时**：
   - 从文件中提取 $C_{key}$ 和IV
   - 使用RSA私钥解密：$K_{SM4} = RSA_{dec}(pri\_key, C_{key})$
   - 使用 $K_{SM4}$ 和IV以SM4-CBC-CTS模式解密密文

这种设计使得：
- 文件内容使用高速的SM4加密（适合大文件）
- 会话密钥使用RSA保护（仅加密16字节，速度快）
- 每次加密使用不同的随机会话密钥（前向安全性）
- 会话密钥随密文文件一起存储（无需额外密钥管理）

## 4.3 加密文件格式

加密后的 `.enc` 文件采用以下二进制格式：

```
┌────────────┬───────────────┬────────────┬──────────────┬───────────┬───────┬──────────────┐
│  MAGIC(8)  │enc_key_len(4)│ iv_len(4)  │ enc_key(var) │  IV(16)   │flags(1)│ ciphertext   │
│"SM4ENC02"  │  uint32 BE   │ uint32 BE  │  RSA加密的   │随机初始化 │模式标识│ SM4-CBC-CTS  │
│            │              │            │  SM4会话密钥  │向量      │       │ 密文数据      │
└────────────┴───────────────┴────────────┴──────────────┴───────────┴───────┴──────────────┘
```

**各字段说明**：

- **MAGIC (8字节)**：固定标识字符串 `"SM4ENC02"`，用于识别文件格式。其中 `"02"` 版本号表示支持CTS模式。

- **enc_key_len (4字节)**：大端序32位无符号整数，表示随后的RSA加密密钥的字节长度。对于RSA-2048，此值固定为256。

- **iv_len (4字节)**：大端序32位无符号整数，表示随后IV的字节长度。本系统固定为16（SM4分组长度）。

- **enc_key (变长)**：使用RSA公钥以OAEP填充加密的SM4会话密钥（128 bit），加密后长度由RSA密钥长度决定（RSA-2048产生256字节）。

- **IV (16字节)**：随机生成的初始化向量，用于CBC模式的初始异或操作。

- **flags (1字节)**：模式标识字节。$0\text{x}01$ 表示CTS模式（明文 ≥ 16字节），$0\text{x}00$ 表示PKCS#7填充模式（明文 < 16字节）。该字段确保解密方能准确选择解密策略。

- **ciphertext (变长)**：SM4-CBC-CTS加密产生的密文数据，长度等于明文长度（CTS模式）或16字节（PKCS#7模式）。

**文件头总大小**：$8 + 4 + 4 + 256 + 16 + 1 = 289$ 字节（RSA-2048情况下）。

## 4.5 流式处理策略

对于大文件，一次性将全部明文/密文读入内存是不可行的（例如1 GB文件将占用1 GB内存）。本系统采用**分段流式处理 + CTS尾块延迟处理**的策略：

### 加密流式处理：

1. **Phase 1（标准CBC流）**：将前 $(q-1) \times 16$ 字节的明文以16字节对齐的块为单位，通过OpenSSL EVP接口进行标准CBC加密，边读边加密边写。此阶段内部维护 `prev_ct[16]` 追踪最后一个密文块，用于后续CTS。

2. **Phase 2（CTS尾块处理）**：从输入文件读取最后两个部分（$P_q$ 的完整16字节 + $P_{q+1}^*$ 的 $r$ 字节），应用CBC-CS1算法中的CTS步骤，执行两次裸SM4块加密和重组，输出最后的密文。

### 解密流式处理：

1. **Phase 1**：将前 $ct\_len - 16 - r$ 字节的密文通过EVP进行标准CBC解密。

2. **Phase 2**：读取 $C_{last}$（16字节）和 $C_{temp}^*$（$r$ 字节），应用CTS解密步骤，恢复 $P_q$ 和 $P_{q+1}^*$。

这种设计使得内存占用始终为 $\mathcal{O}(BUF\_SIZE)$（约4 KB），不随文件大小增长。

# 五、关键实现技术

本章详细分析系统核心模块的C语言实现。代码使用OpenSSL的EVP（Envelope）高级API访问SM4算法，通过GTK 3框架构建图形界面。

## 5.1 单块SM4加解密

为了在CBC-CTS的Phase 2中手动实现密文挪用的加密和解密逻辑，需要绕过OpenSSL EVP的CBC自动链接机制，直接执行单块的裸SM4加密/解密操作。

经过实际测试发现，OpenSSL 3.x版本的 `EVP_sm4_ecb()` 在解密方向返回0字节输出（可能是一个兼容性问题）。因此本系统采用**CBC模式配合零IV**的方式获取得与ECB等效的单块加密效果：

```c
static int sm4_raw_encrypt_block(const unsigned char key[16],
                                 const unsigned char in[16],
                                 unsigned char out[16]) {
    EVP_CIPHER_CTX *ctx = EVP_CIPHER_CTX_new();
    if (!ctx) return 0;
    int outlen = 0, ret = 1;
    unsigned char zero_iv[16] = {0};
    /* 使用CBC+零IV替代ECB（某些OpenSSL版本的ECB有兼容性问题） */
    if (EVP_EncryptInit_ex(ctx, EVP_sm4_cbc(), NULL, key, zero_iv) != 1) ret = 0;
    if (ret) EVP_CIPHER_CTX_set_padding(ctx, 0);  /* 禁用填充 */
    if (ret && EVP_EncryptUpdate(ctx, out, &outlen, in, 16) != 1) ret = 0;
    if (ret && outlen != 16) ret = 0;
    EVP_CIPHER_CTX_free(ctx);
    return ret;
}
```

**技术要点分析**：

- **CBC+零IV等价性**：当IV为零向量且仅处理单个块时，CBC模式退化为：$C = E_K(P \oplus 0) = E_K(P)$，与ECB模式数学上等价。

- **禁用填充**：`EVP_CIPHER_CTX_set_padding(ctx, 0)` 是关键的设置——它告知OpenSSL不要自动添加PKCS#7填充，允许我们以16字节为精确单位控制数据流。

- **解密函数**`sm4_raw_decrypt_block`结构与加密函数对称，使用 `EVP_DecryptInit_ex` 初始化。

## 5.2 文件加密实现（含CTS）

`encrypt_file()` 函数是系统的核心，约200行代码，实现了完整的文件加密流程：

```c
static int encrypt_file(const char *in_path, const char *out_path, const char *pub_path,
                        double *seconds, double *mbps, char *err, size_t errlen)
```

### 5.2.1 加密前处理

函数首先执行以下准备工作：

1. **获取文件大小**：通过 `file_size(in_path)` 确定明文长度，据此判断后续处理策略。

2. **生成密钥材料**：使用 `RAND_bytes()` 生成128 bit随机SM4密钥和128 bit随机IV。`RAND_bytes()` 是OpenSSL的密码学安全伪随机数生成器（CSPRNG），基于操作系统的熵源（Linux上的 `/dev/urandom`）。

3. **RSA密钥保护**：调用 `rsa_encrypt_key()` 使用RSA公钥加密SM4会话密钥。

4. **写入文件头**：按4.3节定义的格式依次写入MAGIC、密钥长度、IV长度、加密密钥、IV和flags字节。

### 5.2.2 加密分支策略

根据明文长度分为三种处理路径：

```
if (input_size == 0) {
    // 空文件：仅写头，无密文
} else if (input_size < 16) {
    // 超短文件：PKCS#7填充（密文16字节）
    // flags = 0x00
} else {
    // 正常文件（≥ 16字节）：CBC-CTS（密文 = 明文长度）
    // flags = 0x01 (CTS_FLAG)
}
```

**超短文件回退**：对于明文长度 < 16字节的文件，使用标准SM4-CBC配合OpenSSL默认的PKCS#7填充处理。此时密文始终为16字节（有1~15字节的膨胀）。这个回退仅在极小文件时才触发，实际场景中极少遇到。

### 5.2.3 CBC-CTS加密核心

**Phase 1 — 标准CBC流式加密**：

```c
ctx = EVP_CIPHER_CTX_new();
EVP_EncryptInit_ex(ctx, EVP_sm4_cbc(), NULL, sm4_key, iv);
EVP_CIPHER_CTX_set_padding(ctx, 0);  /* 关键：禁用自动填充 */

unsigned char prev_ct[16];
memcpy(prev_ct, iv, 16);  /* 初始化为IV */

long cbc_bytes = (r == 0) ? input_size : 
                 (q == 1) ? 0 : (q - 1) * 16;

/* 以16字节对齐的块为单位流式加密 */
while (remaining > 0) {
    int chunk = min(remaining, BUF_SIZE);
    nread = fread(inbuf, 1, chunk, fin);
    EVP_EncryptUpdate(ctx, outbuf, &outlen, inbuf, nread);
    fwrite(outbuf, 1, outlen, fout);
    if (outlen >= 16)
        memcpy(prev_ct, outbuf + outlen - 16, 16);  /* 更新prev_ct */
    remaining -= nread;
}
```

**`prev_ct`追踪的作用**：在CBC模式下，每个明文块 $P_i$ 的加密依赖于前一个密文块 $C_{i-1}$。`prev_ct` 变量始终保存最近一次输出的最后16字节密文块，用于Phase 2中CTS的 $C_{q-1}$（最后一个标准CBC密文块）。

**Phase 2 — CTS尾块处理**（$r > 0$ 时）：

```c
/* 读取最后两个部分 */
unsigned char Pq[16], Pr[16] = {0};
fread(Pq, 1, 16, fin);               /* 最后一个完整明文块 */
fread(Pr, 1, r, fin);                /* 不完整明文块（r字节） */

/* C_temp = E_K(Pq XOR prev_ct) */
xor_block(xor_buf, Pq, prev_ct);
sm4_raw_encrypt_block(sm4_key, xor_buf, C_temp);

/* 填充：P_padded = P* || C_temp[0..15-r]  （挪用C_temp的开头字节） */
unsigned char P_padded[16];
memcpy(P_padded, Pr, r);
memcpy(P_padded + r, C_temp, 16 - r);   /* 挪用！ */

/* C_last = E_K(P_padded XOR C_temp) */
xor_block(xor_buf, P_padded, C_temp);
sm4_raw_encrypt_block(sm4_key, xor_buf, C_last);

/* 输出：C_last（全16字节） || C_temp[0..r-1]（截断至r字节） */
fwrite(C_last, 1, 16, fout);
fwrite(C_temp, 1, r, fout);
```

**关键代码注释**：$C_{temp}$ 的16字节中，前 $(16-r)$ 字节被"挪用"去填充 $P_{q+1}^*$，后 $r$ 字节被丢弃。然而，$C_{temp}$ 的前 $r$ 字节仍作为最终输出的一部分。这种设计使得被挪用的字节恰好是出中不会被输出的字节。**整个加密文件与明文文件大小完全相同——这就是"零膨胀"密码学的工程实现。**

## 5.3 文件解密实现（含CTS逆过程）

`decrypt_file()` 函数与加密函数对称，实现了完全逆向的CTS解密。

### 5.3.1 解密前处理

1. **验证MAGIC**：读取前8字节，与 `"SM4ENC02"` 比对，确保文件格式正确。

2. **读取头部字段**：依次读取 `enc_key_len`、`iv_len`、"加密密钥"、IV、flags字节。

3. **RSA解密密钥**：使用RSA私钥解密出SM4会话密钥。

4. **确定密文大小**：$ct\_size = file\_size - header\_size$（其中 $header\_size = 8 + 4 + 4 + enc\_key\_len + 16 + 1 = 289$）。

### 5.3.2 解密分支策略

```
if (flags & 0x01) {
    // CTS模式 → 根据 r = ct_size % 16 判断是否需要CTS
} else {
    // PKCS#7模式 → 标准CBC解密
}
```

### 5.3.3 CBC-CTS解密核心

**Phase 2 — CTS尾块解密**（$r > 0$ 时）：

这是CTS-1解密算法的核心实现，包含了密文块的重建过程：

```c
/* 读取CTS密文尾块 */
unsigned char C_last[16], C_temp_star[16] = {0};
fread(C_last, 1, 16, fin);          /* 交换块（全16字节） */
fread(C_temp_star, 1, r, fin);      /* 截断块（仅r字节） */

/* Step 1: D_last = D_K(C_last) = P_padded XOR C_temp */
unsigned char D_last[16];
sm4_raw_decrypt_block(sm4_key, C_last, D_last);

/* Step 2: 恢复短块明文 P*[i] = D_last[i] XOR C_temp_star[i] */
unsigned char P_star[16] = {0};
for (int i = 0; i < r; i++)
    P_star[i] = D_last[i] ^ C_temp_star[i];

/* Step 3: 顺序递推恢复完整C_temp */
unsigned char C_temp_full[16];
memcpy(C_temp_full, C_temp_star, r);  /* 已知的前r字节 */
for (int i = r; i < 16; i++)
    C_temp_full[i] = D_last[i] ^ C_temp_full[i - r];

/* Step 4: 解密倒数第二个完整块 */
unsigned char D_temp[16];
sm4_raw_decrypt_block(sm4_key, C_temp_full, D_temp);
unsigned char Pq[16];
xor_block(Pq, D_temp, prev_ct);  /* Pq = D_K(C_temp) XOR C_{q-1} */

/* 输出：Pq（16字节） || P_star（r字节） */
fwrite(Pq, 1, 16, fout);
fwrite(P_star, 1, r, fout);
```

**顺序递推恢复的正确性证明**：

对于 $i \in [r, 15]$，已知 $D_{last}[i] = C_{temp}[i-r] \oplus C_{temp}[i]$（根据3.3.4节的分析），因此：

$$C_{temp}[i] = D_{last}[i] \oplus C_{temp}[i-r]$$

递推过程：
- $i = r$ 时，$C_{temp}[r]$ 依赖于 $C_{temp}[0]$，后者已知（来自 $C_{temp}^*$）✓
- $i = r+1$ 时，$C_{temp}[r+1]$ 依赖于 $C_{temp}[1]$，后者已知 ✓
- ...
- $i = 2r-1$ 时，$C_{temp}[2r-1]$ 依赖于 $C_{temp}[r-1]$，后者已知 ✓
- $i = 2r$ 时，$C_{temp}[2r]$ 依赖于 $C_{temp}[r]$，后者在步骤 $i=r$ 中已恢复 ✓
- 以此类推，所有16字节均可唯一确定 ✓

该递推的数学结构是一个**三角形线性方程组**，具有唯一解。

## 5.4 RSA密钥管理

### 5.4.1 密钥对生成

```c
static int generate_rsa_keys(const char *pub_path, const char *pri_path,
                             char *err, size_t errlen) {
    RSA *rsa = RSA_new();
    BIGNUM *bn = BN_new();
    BN_set_word(bn, RSA_F4);           /* 公钥指数 e = 65537 */
    RSA_generate_key_ex(rsa, 2048, bn, NULL);  /* 生成2048位RSA密钥对 */

    /* 写入私钥文件 */
    PEM_write_RSAPrivateKey(fp, rsa, NULL, NULL, 0, NULL, NULL);

    /* 写入公钥文件 */
    PEM_write_RSAPublicKey(fp, rsa);
}
```

密钥以PEM（Privacy Enhanced Mail）格式存储，私钥不加密保护（简化设计，实际应用中应设置密码短语）。

### 5.4.2 SM4密钥保护

加密侧使用 `RSA_public_encrypt()` 配合 `RSA_PKCS1_OAEP_PADDING` 选项加密SM4会话密钥。解密侧使用 `RSA_private_decrypt()` 恢复。OAEP填充确保了：
- 输出的加密密钥长度固定为256字节（RSA-2048）
- 每次加密产生不同的密文（OAEP引入随机种子）

### 5.5.1 速度统计实现

加密和解密均使用C标准库的 `clock()` 函数进行高精度计时：

```c
start = clock();
/* ... 加密/解密操作 ... */
end = clock();

*seconds = (double)(end - start) / CLOCKS_PER_SEC;
*mbps = ((double)input_size / 1024.0 / 1024.0) / (*seconds);
```

计时范围仅包括纯加密/解密操作的时间，不包括RSA密钥操作和文件I/O的初始开销。`CLOCKS_PER_SEC` 通常为1000000（微秒级精度），足以满足性能测试需求。

2. **CTS与标准CBC性能无差异**：当文件大小为16的倍数时，CTS不执行额外操作，性能与标准CBC一致；当文件存在短块时，CTS仅对最后两个块执行额外的两次裸SM4操作（共32字节的额外处理），相对于整个文件的处理量可忽略不计。

3. **加密与解密速度基本对称**：SM4算法的加解密结构高度对称（仅轮密钥顺序不同），因此加密和解密的吞吐量基本一致。

### 8.3.4 理论吞吐量估算

以SM4单块加密时间 $t_{block}$ 为例，理论吞吐量为：

$$Throughput = \frac{16}{t_{block}} \text{ bytes/s}$$

对于软件实现的SM4，$t_{block}$ 通常在数十至数百纳秒级别，理论吞吐量可达数百MB/s。实际性能受以下因素制约：

- OpenSSL库的实现效率
- 虚拟机的I/O性能
- 文件系统的缓存策略
- CPU时钟频率和缓存命中率

# 九、安全性分析

## 9.1 SM4算法的安全性

SM4作为国家密码标准，其安全性在公开发表后已得到充分的分析和验证：

- **密钥空间**：128 bit密钥空间为 $2^{128}$，暴力破解在计算上不可行。
- **已知最优攻击**：对全轮SM4的最优密码分析攻击仅能覆盖23轮（不可能差分攻击）或23轮（线性攻击），远低于32轮的全轮数。这意味着SM4拥有约9轮的"安全冗余"。
- **侧信道防护**：SM4的结构对时间侧信道具有天然的抵抗力——加密路径不依赖于密钥或明文的值。

## 9.2 CBC模式的安全性

CBC模式的安全性特征：

- **语义安全性**：在IV随机且不可预测的前提下，CBC模式提供IND-CPA安全（选择明文攻击下的不可区分性）。
- **错误传播**：解密时1 bit的密文错误导致当前块完全损坏且下一个块的对应比特翻转。这一特性可用于检测篡改，但不能替代消息认证码（MAC）。
- **IV随机性要求**：本系统使用 `RAND_bytes()` 生成密码学安全的随机IV，满足CBC安全性的前置条件。

## 9.3 CTS的安全性

CTS是CBC模式的扩展变体，其安全性特征如下：

- CTS**不引入额外的安全弱点**：CTS仅改变最后两个块的加密方式，不涉及额外的密钥材料或随机数。NIST SP 800-38A Addendum已对CTS模式进行了标准化和安全性分析。
- CTS保持了CBC的IND-CPA安全属性：只要底层分组密码是伪随机置换（PRP），CBC-CTS就保持IND-CPA安全。
- CTS不提供认证功能：与CBC一样，CTS是保密性模式而非认证加密模式。需要数据完整性保护时，应结合HMAC或使用GCM等认证加密模式。

## 9.4 RSA密钥保护的安全性

- **密钥长度**：2048 bit RSA密钥提供了约112 bit的安全强度，与SM4的128 bit安全强度基本匹配（NIST SP 800-57）。
- **OAEP填充**：RSA-OAEP是RSA的最优非对称加密填充方案，在随机预言机模型下可证明达到IND-CCA2安全。
- **会话密钥随机性**：每次加密使用全新的随机128 bit SM4会话密钥，提供前向安全性（一次密钥泄露仅影响该次加密的文件）。

## 9.5 系统的安全局限性

本系统作为一个教学性质的实验项目，存在以下安全局限：

1. **缺乏认证机制**：系统仅提供加密功能，不包含消息认证码（MAC）或数字签名。攻击者可以篡改密文而无法被检测（但篡改后的解密结果将是随机的）。

2. **私钥存储安全**：私钥以无保护的PEM格式存储在磁盘上。在生产环境中，私钥应使用密码短语加密存储或使用硬件安全模块（HSM）。

3. **无密钥管理机制**：系统不提供密钥轮换、密钥撤销、公钥基础设施（PKI）等密钥生命周期管理功能。

4. **文件完整性**：建议在实际应用中结合SHA-256等哈希算法对原始文件计算摘要并加密存储。

# 十、实验总结

## 10.1 实验成果

本实验成功设计并实现了一个基于SM4分组密码算法的文件加密软件系统，主要成果包括：

1. **完整的加解密功能**：系统实现了文件的加密和解密，支持任意大小和任意类型的文件。

2. **密文链接（CBC模式）**：系统采用CBC工作模式，每个明文块在加密前与前一密文块异或，实现了块间的"链接"效应，避免了ECB模式的安全缺陷。

3. **密文挪用（CTS短块处理）**：系统实现了CBC-CS1密文挪用技术，对于≥16字节的文件，密文长度严格等于明文长度（零膨胀），是区别于传统PKCS#7填充的关键技术特征。通过29个边界测试用例的验证，CTS算法实现完全正确。

4. **速度统计功能**：系统使用`clock()`函数对加密和解密操作进行计时，计算并显示MB/s吞吐率，为用户提供了直观的性能反馈。

5. **RSA混合加密**：采用RSA-2048 OAEP加密SM4会话密钥，解决了对称密钥的安全分发问题。

6. **图形化人机界面**：基于GTK 3框架，提供了文件选择、密钥生成、加密/解密操作和日志显示等功能，使用简单直观。

## 10.2 关键技术贡献

本实验最核心的技术贡献在于**CBC-CTS（密文链接+密文挪用）短块处理技术**的完整实现。本文给出的CTS加密和解密算法、顺序递推恢复算法以及流式处理策略，构成了一个完整的工程解决方案。主要技术创新点包括：

1. **CTS-1顺序递推恢复算法**：通过分析$D_{last} = P_{padded} \oplus C_{temp}$的逐字节等式关系，发现$C_{temp}$的未知字节可以通过顺序递推唯一恢复，无需暴力枚举。

2. **流式+CTS双层处理架构**：Phase 1使用EVP流式CBC处理主体数据，Phase 2使用裸SM4块操作处理CTS尾块，兼顾了内存效率和算法正确性。

3. **CBC+零IV等效ECB技巧**：针对某些OpenSSL版本ECB解密的兼容性问题，使用CBC模式配合零IV实现等效的单块裸加密/解密。

## 10.3 实验收获

通过本实验，对以下知识点有了深入的理解：

1. **分组密码工作模式的设计原理**：深入理解了CBC模式如何通过反馈机制提高安全性，以及不同模式之间的安全性差异。

2. **短块处理技术的工程实现**：掌握了CTS的关键思想——利用已产生的密文块填充短块，在不引入额外数据的前提下实现任意长度加密。

3. **OpenSSL EVP API的使用**：熟练掌握了OpenSSL高级加密接口的编程方法，包括上下文管理、填充控制、模式选择等。

4. **混合密码体制的工程实践**：理解了如何在实际系统中结合对称和非对称密码各自的优势。

5. **GTK图形界面编程**：掌握了GTK 3的事件驱动编程模型和基本控件的使用方法。

6. **软件测试方法论**：通过对29个边界情况的系统性测试，实践了分区测试、边界值测试等软件测试方法。

## 10.4 改进方向

未来可从以下方向改进系统：

| 方向 | 具体措施 |
|------|---------|
| **认证加密** | 引入HMAC-SHA256或直接使用GCM模式，实现保密+认证 |
| **密钥派生** | 使用PBKDF2/bcrypt从用户口令派生加密密钥 |
| **多线程** | 对大文件使用多线程并行化SM4加密 |
| **文件格式标准化** | 遵循PKCS#7/CMS等标准封装加密数据 |
| **硬件加速** | 利用Intel AES-NI等效的SM4硬件指令 |
| **密钥管理** | 集成PKI和证书管理 |
